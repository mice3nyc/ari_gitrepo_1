# Iteration Protocol

> 피드백을 목차 변경으로 전환하는 지시문.
> 이 프로토콜이 프레임워크의 심장이다 — 피드백을 받아서 실제로 고치는 단계.

---

## 입력

- `feedback/round_N_feedback.md` — 이번 라운드의 피드백
- `prototype/toc_vN.md` — 현재 버전의 목차

## 출력

- `prototype/toc_v(N+1).md` — 개선된 목차
- `rounds/round_N/changes.md` — 이번 라운드의 변경 기록
- `rounds/round_N/diff.md` — vN과 v(N+1)의 차이

---

## 단계

### Step 1: 피드백 우선순위 정렬

피드백의 "가장 심각한 발견 TOP 3"에서 시작한다.
각 발견에 대해:

1. **이 문제가 구조적인가, 표면적인가?**
   - 구조적 = 챕터 순서, 파트 재편, 챕터 분할/병합
   - 표면적 = 제목 수정, 설명 보강, 용어 정리

2. **변경의 파급 범위는?**
   - 단일 챕터 내 수정
   - 복수 챕터에 영향
   - 파트 전체 재구성

3. **변경 후 다른 시나리오에서 악화되지 않는가?**
   - 시나리오 A의 피드백을 반영했을 때 시나리오 B가 나빠지면, 둘 다 기록하고 트레이드오프를 명시

### Step 2: 변경 계획 작성

`rounds/round_N/changes.md`에 기록:

```
## Round N 변경 계획

### 변경 1: [제목]
- 근거: 피드백 #N-N
- 유형: 구조적 / 표면적
- 변경 내용: 구체적 서술
- 파급 범위: 영향받는 챕터 목록
- 트레이드오프: 있다면

### 변경 2: ...
```

### Step 3: 목차 수정 실행

- `toc_vN.md`를 복사하여 `toc_v(N+1).md` 생성
- 변경 계획에 따라 수정
- **원칙**:
  - 한 라운드에 구조적 변경은 최대 3건
  - 과도한 변경은 다음 라운드로 이월
  - 변경하지 않기로 한 피드백도 기록 ("이유: ...")

### Step 4: Diff 생성

`rounds/round_N/diff.md`에 기록:

```
## Round N Diff: v(N) → v(N+1)

### 추가된 챕터
-

### 삭제된 챕터
-

### 이동된 챕터
- Ch.X: Part Y → Part Z

### 분할된 챕터
- Ch.X → Ch.Xa + Ch.Xb

### 병합된 챕터
- Ch.X + Ch.Y → Ch.Z

### 제목 변경
- "기존 제목" → "새 제목"

### 설명 보강/수정
- Ch.X: 변경 요약
```

### Step 5: 검증

새 목차(toc_v(N+1))에 대해 즉석 검증:
- 변경 근거가 된 피드백이 실제로 해소되었는가?
- 새로 생긴 문제는 없는가? (간단히 3개 시나리오 헤드라인만 체크)
- 전체 챕터 수가 합리적 범위(20~30) 안에 있는가?

---

## 이월 원칙

- 이번 라운드에서 다루지 못한 피드백은 `rounds/round_N/backlog.md`에 기록
- 다음 라운드 시작 시 backlog를 먼저 확인
- 3라운드 이상 이월되는 피드백은 "이 문제가 정말 중요한가?" 재평가

---

## Self-Improvement

이 프로토콜은 라운드를 거치며 다음이 개선되어야 한다:
- "한 라운드 구조적 변경 최대 3건" 규칙이 너무 적거나 많으면 조정
- 트레이드오프 판단 기준이 모호하면 구체화
- Diff 형식이 실제로 유용한지 (너무 장황하거나 너무 간략하면 수정)
- 검증 단계가 형식적으로 되지 않도록 — 실제로 문제를 잡아낸 사례를 기록하여 강화
- Step 3의 "변경하지 않기로 한 이유"가 가장 중요한 학습 자료 — 이것이 축적되면 "이 목차의 설계 원칙"이 자연스럽게 드러난다
