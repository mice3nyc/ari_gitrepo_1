# Framework Rules

> 이 프레임워크 자체의 규칙.
> PX 책의 Ch.12(규칙의 설계)가 말하는 바로 그것 —
> "규칙은 제한이 아닌 가능성의 조건이다."

---

## 핵심 규칙

### Rule 1: 원본 보존
`toc_v0.md`는 절대 수정하지 않는다. 모든 개선은 새 버전 파일로.
이유: diff의 기준점이 필요하다. 어디서 출발했는지 모르면 얼마나 왔는지 모른다.

### Rule 2: 한 라운드 = 한 사이클
한 라운드는 반드시 playtest → feedback → iterate → reflect 순서로 완주한다.
중간에 멈추지 않는다. 피드백만 하고 반영 안 하는 건 플레이테스팅이 아니다.

### Rule 3: 구조적 변경 상한
한 라운드에 구조적 변경(챕터 순서, 파트 재편, 분할/병합)은 최대 3건.
이유: 한꺼번에 많이 바꾸면 뭐가 효과를 냈는지 알 수 없다.

### Rule 4: 트레이드오프 명시
시나리오 간 충돌하는 피드백이 있으면, 선택과 그 이유를 기록한다.
"A를 위해 B를 희생했다"가 기록되어야 나중에 재검토할 수 있다.

### Rule 5: 매 라운드 끝에 자기 검사
라운드 종료 시 반드시 `rounds/round_N/reflect.md`를 작성한다.
이 파일에서 규칙, 시나리오, 프로토콜 자체를 검토한다.

### Rule 6: 3라운드 정체 시 파괴
3라운드 연속 의미 있는 변경이 없으면:
- 시나리오를 전면 교체하거나
- 새로운 시나리오를 추가하거나
- 프레임워크 구조 자체를 재검토한다
정체는 "완성"이 아니라 "관점의 고갈"이다.

### Rule 7: 목차 밖의 발견도 기록 [Round 1 강화: 의무화]
라운드 중 목차와 무관한 발견(책의 톤, 독자 정의, 제목 등)은
`rounds/round_N/discoveries.md`에 **반드시** 별도 기록.
목차 개선에만 시야를 좁히지 않는다.
Round 1에서 UX 프레이밍 전략이라는 중요한 목차-밖 발견이 있었으나 기록하지 않았다.

### Rule 8: 인수조건은 매 라운드 끝에 재심사한다 [Round 2 신설]
`rules/acceptance_criteria.md`를 reflect 시점에 검토한다.
"이 인수조건이 프레임워크를 개선하는가, 제약하는가?"를 묻고, 필요하면 v(N+1)을 작성한다.

## 라운드 실행 체크리스트

```
Round N 체크리스트:

- [ ] playtest: 시나리오 01 실행
- [ ] playtest: 시나리오 02 실행
- [ ] playtest: 시나리오 03 실행
- [ ] feedback: round_N_feedback.md 작성
- [ ] feedback: 교차 분석 완료
- [ ] iterate: 변경 계획 작성
- [ ] iterate: toc_v(N+1).md 생성
- [ ] iterate: diff.md 작성
- [ ] reflect: round_N reflect.md 작성
- [ ] rules: 규칙 수정 필요 여부 판단
- [ ] scenarios: 시나리오 수정 필요 여부 판단
```

## Self-Improvement

이 규칙 파일은 라운드를 거치며 다음이 개선되어야 한다:

### 매 라운드 reflect에서 확인할 것:
1. **규칙이 도움이 됐는가, 방해가 됐는가?**
   - "Rule 3 때문에 중요한 변경을 이월해야 했다" → 상한 조정 검토
   - "Rule 6의 3라운드 기준이 너무 길다/짧다" → 조정

2. **빠진 규칙이 있는가?**
   - 라운드 중 암묵적으로 따른 원칙이 있으면 명문화

3. **죽은 규칙이 있는가?**
   - 한 번도 적용되지 않는 규칙은 삭제하거나 이유를 재검토
