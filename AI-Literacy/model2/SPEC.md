## 2. 핵심 메카닉 — v0.1~v0.3 교훈 반영

### 놀공 게임의 공통 패턴 (반드시 따를 것)
> "정보가 콘텐츠 영역에 존재하고, 학생이 탐색하며, 빈칸(보드)에 채운다."

| 원칙 | 설명 | v0.1~v0.3에서 배운 것 |
|------|------|------|
| **빈칸이 먼저** | 보드의 빈칸이 목표로 보여야 한다 | v0.1: 카드 목록이 먼저 → 수동적. v0.2에서 보드 먼저로 개선 |
| **찾는 것이 게임** | 콘텐츠를 탐색하며 "이게 여기 들어가는 거다!"를 발견하는 순간이 핵심 | v0.2: 선택지 3개에서 고르기 → 퀴즈. v0.3: 카드를 넘기며 스스로 판단 → 개선됨 |
| **퀴즈 금지** | O/X, 3지선다, 정답 맞히기 구조는 퀴즈이지 게임이 아니다 | v0.1~v0.2 모두 퀴즈가 됨. 피터공 3회 지적 |
| **화면 분할** | 콘텐츠 탐색 영역과 수집/입력 영역이 분리 | v0.3에서 도입. DMZ 게임 참고 — 소스 읽기 화면과 빈칸 입력 화면이 별도 |
| **수집한 것이 나를 비춘다** | 같은 카드를 보고도 학생마다 다른 보드가 완성된다 | 아직 미구현 — 현재는 정답이 있는 구조. **v0.4에서 해결 필요** |

### 모델2 고유 메카닉: 4가지 신호 수집

```
[보드: 빈칸 4개]          [카드 브라우저: 정보 카드 넘기기]
┌─────────┐ ┌─────────┐   ┌──────────────────────┐
│🔵 기술   │ │🟡 사회   │   │  ← 카드 3/12 →        │
│ (빈칸)   │ │ (빈칸)   │   │                      │
├─────────┤ ├─────────┤   │  [기사/인터뷰/데이터]   │
│🔴 개인   │ │🟣 미래   │   │  제목, 본문, 출처      │
│ (빈칸)   │ │ (빈칸)   │   │                      │
└─────────┘ └─────────┘   │  [이 카드를 등록 ▼]    │
                          └──────────────────────┘
```

학생이 하는 행동:
1. 카드를 넘기며 읽는다 (기사, 인터뷰, 데이터, SNS 등 다양한 형식)
2. "이 카드는 기술 신호에 해당한다"고 판단하면 → 보드의 기술 슬롯에 등록
3. 4개 슬롯을 다 채우면 → 조립 → 브리핑 → 거울

## 3. 해결해야 할 핵심 문제 (v0.4)

### 문제 1: "정답이 있는 분류" → 퀴즈가 된다
v0.3에서 각 카드에 `fits:['tech']` 같은 정답이 있다. 맞으면 통과, 틀리면 "다른 신호에 가깝습니다".
이건 결국 분류 퀴즈다.

**해결 방향**: 대부분의 카드가 **여러 신호에 해당할 수 있어야** 한다.
- "콜로라도 미술대회 AI 수상" 카드 → 기술로도, 사회로도 등록 가능
- 학생의 선택에 따라 보드 구성이 달라진다 = 거울
- **순수 노이즈(출처 불명, 과장)만 거부**, 나머지는 어디에 넣어도 수용

### 문제 2: 개인/미래 신호의 처리
기술/사회는 팩트 기반 카드에서 찾지만, 개인/미래는 학생의 판단이다.

**해결 방향**:
- 개인 신호: 모든 카드를 읽은 뒤, "이 이슈가 나에게 의미하는 건?" → **조각 선택** (3개 중 택1, 정답 없음)
- 미래 신호: "5년 후에는?" → **시나리오 조각 2개를 골라 연결** (조합에 따라 다른 시나리오)
- 이 둘은 카드 브라우저에서 찾는 것이 아니라, 보드 슬롯 클릭 시 **별도 인터랙션**으로 처리

### 문제 3: "찾는 느낌"의 강화
v0.3에서 카드 넘기기 + 등록 버튼은 개선됐지만, DMZ의 "아, 이게 저기 빈칸에 들어가는 거구나!" 감각은 아직 약하다.

**해결 방향**:
- 카드 본문에서 핵심 정보를 **볼드로 강조** → 읽다가 "이 부분이 핵심이구나" 인지
- 보드 슬롯에 **구체적인 질문**을 표시 → "이 사건에서 AI 기술은 구체적으로 무엇을 했나?" → 카드를 읽다가 해당 답을 발견
- 등록 시 카드 전체가 아니라 **핵심 문장이 보드에 표시** → 찾은 것이 명확히 보임
- 등록 후 **보드 슬롯이 시각적으로 채워지는 애니메이션** → 보상감

## 4. 화면 구조

### 4.1 인트로
- 역할 부여: "당신은 시그널 에이전트입니다"
- 미션 카드: 이슈 제목 + 간단 설명
- 시작 버튼

### 4.2 메인 게임 (분할 화면)

### 레이아웃 원칙
- **전체 앱을 브라우저 중앙에 고정 폭 컨테이너로 배치** (max-width: 720px, 센터 정렬)
- 요소가 브라우저 벽면으로 몰리지 않도록 충분한 패딩(20~24px)
- 분할 화면도 중앙 컨테이너 안에서 구성

**PC (960px+)**:
```
              ┌────────────────────────────────┐
              │ SIGNAL AGENT              2/4  │
              ├────────┬───────────────────────┤
              │        │                       │
              │ BOARD  │   CARD BROWSER        │
              │ 240px  │   (나머지 폭)          │
              │        │                       │
              │ 🔵 기술 │   ← 3/12 →           │
              │ 🟡 사회 │   [카드 내용]          │
              │ 🔴 개인 │                       │
              │ 🟣 미래 │   [등록 버튼 바]       │

**모바일 (700px 이하)**:

          │  CARD BROWSER        │

          │  [등록 버튼 바]       │
          └──────────────────────┘
                ↑ 중앙 ↑
```

### 4.3 카드 브라우저
- **네비게이션**: ← → 화살표로 카드 넘기기 (스와이프도 지원)
- **카드 표시**: 유형 배지(기사/인터뷰/데이터/SNS) + 제목 + 본문(볼드 강조) + 출처
- **등록 바**: 하단에 4개 신호 버튼. 이미 채워진 슬롯은 비활성
- **노이즈 피드백**: 노이즈 카드 등록 시 인라인 경고 (모달 아님)

### 4.4 보드 슬롯
- 빈 상태: 점선 테두리 + 질문 텍스트
- 채워진 상태: 실선 테두리 + 등록된 카드 제목 + ✓
- 개인/미래 슬롯: 클릭 시 별도 선택 UI (카드 브라우저와 별도)

### 4.5 조립 (Assemble)
- 4개 슬롯이 다 채워지면 활성화
- **메인 신호 선택**: "이 이슈에서 가장 중요한 신호는?" → 4개 중 택1
- **헤드라인 선택**: 4개 후보 중 택1
- 조립 완료 → 브리핑 생성

### 4.6 브리핑 (Briefing)
- 한 장의 카드 형태
- 구성: 헤드라인 + 4가지 신호 요약 (메인 신호 강조) + 학생이 고른 미래 시나리오

### 4.7 거울 (Mirror)
- 프로필 유형: 메인 신호 기반 (기술 중심형 / 사회 관찰형 / 개인 체감형 / 미래 설계형)
- 프로필 설명: 1~2문장
- 판정 없음: 어떤 유형이 좋다/나쁘다 없음
- "다시 하기" 버튼

## 5. 데이터 구조

### 5.1 슬롯 (4개)
```javascript
{
  id: 'tech',           // 고유 ID
  label: 'TECH 기술',    // 표시명
  color: '#2563EB',     // 컬러
  question: '이 사건에서 AI 기술은 구체적으로 무엇을 했나?',  // 빈칸 질문
  type: 'card'          // 'card' = 카드에서 찾아 등록 / 'choice' = 선택지에서 고름
}
```

기술, 사회: `type: 'card'` — 카드 브라우저에서 찾아 등록
개인, 미래: `type: 'choice'` — 슬롯 클릭 시 선택 UI

### 5.2 정보 카드
```javascript
{
  id: 1,
  type: 'article',       // article / interview / data / sns
  badge: '기사',          // 표시 배지
  title: '카드 제목',
  body: '본문 (<b>볼드</b>로 핵심 강조)',
  source: '출처, 날짜',
  assignableTo: ['tech', 'social'],  // 등록 가능한 슬롯 (복수 가능)
  isNoise: false,                    // true면 어디에도 등록 불가
  noiseReason: ''                    // 노이즈인 이유
}
```

**핵심**: `assignableTo`가 복수 → 같은 카드도 학생에 따라 다른 슬롯에 들어감

### 5.3 개인 신호 선택지
```javascript
{
  id: 'p1',
  text: '선택지 텍스트',
  // 정답 없음. 어떤 것을 골라도 유효
}
```

### 5.4 미래 신호 조각
```javascript
{
  id: 'f1',
  text: '시나리오 조각 텍스트',
  // 3개 중 2개 선택 → 연결
}
```

## 7. 인터랙션 상세

### 카드 등록 흐름
```
카드 읽기 → 하단 등록 바에서 신호 버튼 클릭
  → assignableTo에 해당 슬롯 포함?
    → YES: 보드에 등록. 토스트 "기술 신호 수집 완료". 슬롯 채워짐 애니메이션
    → NO (다른 신호에 가까움): 인라인 피드백 "이 카드는 다른 신호에도 해당할 수 있어요. 한번 더 생각해보세요."
    → NO (노이즈): 인라인 피드백 "⚠ 이 정보는 출처가 불분명합니다. [이유]"
```

### 개인 신호 등록 흐름
```
보드에서 개인 슬롯 클릭 (또는 기술/사회 다 채운 후 자동 표시)
  → 관련 기사 2개 표시 (배경 읽기용)
  → 3개 선택지 표시
  → 하나 선택 → 보드에 등록
```

### 미래 신호 등록 흐름
```
보드에서 미래 슬롯 클릭 (또는 개인까지 채운 후 자동 표시)
  → 관련 기사 2개 표시 (배경 읽기용)
  → 3개 시나리오 조각 표시
  → 2개 선택 → 연결 문장 자동 생성 → 보드에 등록
```

### 노이즈 카드 처리
- 등록 시도 시 **인라인 경고** (모달 아님, 카드 하단에 표시)
- 경고 후 다른 카드로 넘어갈 수 있음
- 노이즈 카드 자체에는 "노이즈" 라벨이 없음 — 학생이 스스로 판단해야
- 거울 단계에서 "이런 노이즈도 있었습니다" 공개

## 8. 디자인 규칙

### 컬러
- 배경: #FFFFFF
- 텍스트: #111111
- 보더: #111111
- 연한 배경: #F5F5F5
- 비활성: #CCCCCC, #999999
- 포인트 컬러: 신호 4색만 (파랑, 노랑, 빨강, 보라)
- 성공: #16A34A (녹색)
- 탁한 색 절대 금지

### 타이포그래피
- UI 레이블: 모노스페이스, 9~11px, 대문자, letter-spacing 1px
- 본문: 시스템 산세리프, 13~14px
- 제목: 시스템 산세리프, 15~17px, bold
- 카드 제목: 15px bold

### 스페이싱
- 보더 radius: 4px
- 카드 패딩: 12~16px
- 요소 간 간격: 8~12px

### 애니메이션
- 슬롯 채워질 때: 배경 플래시 (0.3s)
- 카드 전환: 좌우 슬라이드 (0.2s)
- 토스트: 하단에서 올라와 2초 후 사라짐

---

## 9. 빌드 전 체크 (canon/constitution.md 기반)

프로토타입 완성 후 반드시 확인:

- [ ] **선택의 동기가 있는가?** — 카드를 넘기는 행위에 목적이 있는가 (빈칸을 채우기 위해)
- [ ] **읽는 건가 하는 건가?** — 카드를 수동적으로 읽기만 하는 구조가 아닌가
- [ ] **레슨을 주고 있는가?** — 결론 문장, 교훈, "~해야 합니다" 톤이 없는가
- [ ] **퀴즈가 아닌가?** — O/X, 정답 맞히기 구조가 아닌가
- [ ] **거울이 작동하는가?** — 학생마다 다른 보드/브리핑이 나오는가
- [ ] **15~20분에 끝나는가?**
- [ ] **AI 냄새가 나지 않는가?** — 문장이 자연스러운가

## 10. 루프 운영

### 자율 루프 절차
1. 에이전트가 이 SPEC + Canon 파일을 읽고 프로토타입 구현
2. 위 체크리스트로 자체 검증
3. 문제 발견 시 수정 → 재검증 (1라운드)
4. 3~5라운드 반복 후 최종 후보 2개 제출
5. 피터공 검토

### 라운드 기록
각 라운드 결과를 `runs/run_N.md`에 기록:
- 이번 라운드에서 시도한 것
- 체크리스트 결과
- 병목
- 다음 라운드 수정 계획
