## 4컷→3컷 구조 변경 + 관점 카드 수집 (2026-04-25)

**피터공 관찰**: "4컷의 무게가 같지 않다. 질문은 독립 장면이 아니다. 질문→선택이 핵심 흐름이다. 결과는 안내가 아니라 획득이어야 한다. 기능은 의미를 만든다."

**변경**:
- 4컷(상황|질문|선택|결과) → **3컷(상황+질문 | 선택 | 결과+획득)**
- 질문을 상황에 통합: 만화에서 "고민하는 캐릭터"는 상황 컷 안에 있다
- 결과 패널에 **관점 카드 수집** 추가: 각 선택에서 한 줄짜리 관점을 카드로 획득
- 최종 리포트에 수집한 관점 카드 목록 표시

**디자인 변경**:
- 박스 라운딩 제거 → 직각
- 이미지 자리 → 정사각형 (ChatGPT 생성 이미지 삽입용)

**관점 카드 설계 원칙**:
- "기능은 의미를 만든다. 쓸일이 있으면 의미있다" — 피터공
- 관점 카드 = 이번 선택에서 배운 한 줄. 정보 확장, 어휘 전달도 가능
- 최종 리포트에서 "네가 모은 관점들"로 보여줌

## UI 이터레이션 피드백 반영 (2026-04-25, 3차~5차)

**피터공 피드백 → 반영 순서**:
1. 스탯바: 레이블+수치 가로 배치, 게이지 초록→노랑→빨강 변화, +/- 팝 위치 수정
2. 선택지 힌트 색상: 절약=초록, 사용=주황, 적음=빨강
3. 선택 후 빈 Panel 버그 → Panel 2에 선택 요약 표시
4. 질문→선택지 시각 연결: 질문 박스 + 말풍선 커넥터
5. 피드백 텍스트+수치 인라인: "얻은 것"=보상, "대신"=비용 분리
6. 4컷→3컷: 질문을 상황에 통합, "어떻게 할까?" [다음] 버튼 흐름
7. Panel 2에 정사각 이미지 자리 + 질문+선택지
8. "얻은 것" 수치 팡팡 pop 효과
9. "대신"에 비용 수치(시간/에너지) 명시
10. 직각 박스, 정사각 이미지

**핵심 원칙 (피터공)**:
- "읽을 필요를 설계해야 한다" — 텍스트에 기능을 붙여야 학생이 읽는다
- "기능은 의미를 만든다. 쓸 일이 있으면 의미있다"
- "정답이 있는 선택은 선택이 아니다"

## UI를 4컷 패널 스타일로 변경 (2026-04-25, 1차)

**피터공 피드백**: "읽는 과제 같다. 전체 흐름 진행이 느껴지지 않는다. 선택지와 질문이 디자인상 구별 안 된다."

**변경 내용**:
- 수직 스크롤 텍스트 레이아웃 → 4컷 패널 가로 배열 (tab1.html 스타일)
- 선택지를 패널 옆에 세로 카드 스택으로 분리
- B&W 볼드 테두리 + box-shadow 리트로 미학 (Paperlogy 폰트)
- 스탯 바에 숫자 펄스/게이지 애니메이션
- 패널 슬라이드인 + 선택지 페이드인/아웃 애니메이션
- 시나리오 패널이 누적되는 만화 스크롤 구조
- Act 사이 바인더 구분선

**보존한 것**: 게임 엔진(상태관리, 선택처리, 레벨업, 스킬보너스), 저장(localStorage), 이벤트 로그(trackEvent), 디버그 패널. 코드 섹션 3-8 동일.

**핵심 원칙**: "게임으로 판단 불가하면 UI/UX 문제" — 피터공. 콘텐츠가 아니라 경험의 형태를 바꿨다.

## 왜 단일 HTML + 바닐라 JS인가

- 빌드 도구 없이 브라우저에서 바로 실행 가능
- 교육 현장에서 파일 하나만 배포하면 됨
- 원본 명세서 섹션 8-9에 React/TS/Vite 권장이 있었으나, 섹션 28의 "단일 HTML, 빌드 과정 없이 브라우저에서 열 수 있어야" 조건과 모순. 클라이언트 전용 원칙에 따라 단일 HTML로 통일

## 왜 서버 없이 시작하는가

- 1차 프로토타입은 핵심 루프 검증이 목적
- 서버 의존성은 배포/테스트 장벽
- 추후 로그/결과만 서버로 전송하면 됨
- trackEvent()를 미리 분리하여 나중에 fetch만 추가

## 왜 실제 AI API를 호출하지 않는가

- 프로토타입은 "AI 사용 시뮬레이션"이 아니라 "위임 판단 경험"
- API 비용/지연/인증이 교육 현장에서 장벽

## 왜 선택지에 정답을 만들지 않는가

- "정답이 있는 선택은 선택이 아니다" — 피터공
- 각 선택은 상황에 따라 합리적. 트레이드오프가 핵심
- "AI를 쓰지 마라"도 "AI를 써라"도 아닌, 판단의 근육을 기르는 경험

## 왜 trackEvent()를 미리 분리하는가

- 현재는 console + localStorage만
- 서버가 생기면 함수 내부에 fetch만 추가
- 분석 데이터 구조를 초기부터 확정

## 플레이테스트 확인 필요 사항

### ISSUE-1: 숫자 밸런스에서 정답이 보이는 문제

**현상**: "맡길 일" 선택은 항상 시간/에너지 소모 낮음 + 경험치 낮음. "내 생각/다시 보기"는 항상 소모 높음 + 경험치 높음. 두 시나리오만 지나면 "직접 하는 게 점수 많이 준다"를 학생이 눈치챈다.

**위험**: 트레이드오프가 아니라 최적화 문제가 됨. 기획문서의 "각 선택지는 나름대로 합리적이어야 한다" 원칙이 수치 레벨에서 깨짐.

**제안된 해법**:
- 자원 고갈 시 제약 도입 (시간 부족 → 빠른 선택만 가능)
- "맡길 일" 선택이 자원 보존 → 후반에 유리한 구조
- 시간/에너지 고갈 시 강제로 AI에게 맡길 수밖에 없는 상황

**플테 확인 질문**: "3번째 시나리오쯤 학생이 '항상 직접 하는 게 낫다'고 느끼는가?"

**결정 시점**: 플레이테스트 후

### ISSUE-2: 6개 시나리오로 성장감 발생 어려움

**현상**: 레벨업 임계값 Lv.2=30, Lv.3=70. 세 스킬을 골고루 선택하면 6시나리오 끝나도 어떤 스킬도 Lv.2에 못 닿을 수 있음.

**수학적 분석**:
- S03의 잠금 선택지는 delegateTask Lv.2 필요 → S01+S02에서 delegateTask만 골라도 10+8=18 < 30. **수학적으로 해금 불가**.
- S05/S06의 잠금 선택지는 reviewResult Lv.2 필요 → 2~3번 연속 reviewResult를 골라야 도달. 가능하지만 빡빡함.
- Lv.3(70)은 6시나리오 내에서 거의 불가능.

**위험**: 성장감 부재 + 잠금 선택지가 영원히 잠김.

**제안된 해법**: 임계값 하향(Lv.2=20), 복수 스킬 XP 증가, 잠금 선택지 Act3 배치

**플테 확인 질문**: "레벨업을 한 번도 경험하지 못하는 플레이가 발생하는가?"

**결정 시점**: 플레이테스트 후. DebugPanel에서 스킬 레벨 조작하여 잠금 해금 테스트 가능.

### ISSUE-3: 자원 고갈의 무의미

**현상**: 시간/에너지가 0이 되어도 아무 일도 안 일어남 (0에서 clamp만).

**위험**: 자원이 점수판일 뿐, 관리의 긴장감 없음.

**제안된 해법**: 시간 10이하 → 빠른 선택만 가능 등의 제약

**플테 확인 질문**: "시간/에너지가 줄어도 긴장감이 없는가?"

**결정 시점**: 프로토타입에서는 clamp만. 플레이테스트 후 제약 추가 여부 결정.

### ISSUE-4: 잠금 선택지 표시 방식

**현상**: "맡길 일 Lv.2 필요"라고 보여주면 "이전에 틀린 선택 했구나" 느낌을 줄 수 있음.

**대안**: 해금 시에만 "새 방법이 생겼다"로 표시.

**프로토타입 구현**: 현재는 잠금 상태 + 이유 표시로 구현 (명세서 기준). 플테 후 대안 검토.
