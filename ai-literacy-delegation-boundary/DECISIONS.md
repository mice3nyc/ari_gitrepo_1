#### 1. 게임 트리 정정 — 5분기 평면 → 1차 ABC + 2차 9 (76 노드)

**기존 (v0.2)**: 시나리오 6개 × 선택지 3개 평면. 위임 강도 라벨로 분기.

**정정**: 시나리오마다 트리 구조.
```
상황 (1)
  → 1차 선택지 (3): A 직접 / B 위임 / C 우회
    → 2차 선택지 (각 1차마다 3 = 9)
      → 결과 (9, raw 결과물 텍스트)
        → 검토 (각 결과당 3 = 27): R1 그대로 / R2 한 번 읽기 / R3 크로스체크
          → 최종 점수 + 획득 아이템 (27)
```

총 76 노드. 학생 1회 플레이 = 3선택(1차→2차→검토)으로 1경로.

**이유**: 위임 강도 라벨(직접/완전위임/얕은협업/학습보조/부분)은 카테고리이지 학생이 화면에서 보는 실제 행동 선택지가 아님. 게임 트리는 학생 행동 단위로 설계되어야.

#### 2. 검토 단계 추가 — 두 역량 게임 단계 분리

**피터공 통찰**: "선택지에 따라 결과가 나오고, 그 결과를 어떻게 평가할 역량이 있느냐 혹은 어떻게 결과를 검토하느냐에 따라 다른 점수의 결과로 끝나는 마지막 검토 단계가 있고 이에 따라 최종 경험치와 획득 아이템이 달라지는 것이지."

**핵심**: 두 역량이 게임 단계로 분리.
- **위임 선택력** = 1차·2차 선택에서 작동
- **지식 (도메인)** = 검토 단계에서 작동
- **최종** = 두 역량 합산 → 점수 + 아이템

**검토 선택지 = 모든 결과 공통 3개**:
- R1. 그대로 제출 (검토 X)
- R2. 한 번 읽고 이상한 곳만 표시 (명시 환각만 발견)
- R3. 출처 확인 + 크로스체크 (미세 오류까지 발견)

**의미성 검증 필요**: R3가 A1(전체 정독) 결과에선 발견할 게 거의 없을 수 있음. 시트 27행 채워봐야 검토 선택지의 결과별 의미성 검증 가능 — 무의미한 경우 솔직히 명시.

## 4컷→3컷 구조 변경 + 관점 카드 수집 (2026-04-25)

**피터공 관찰**: "4컷의 무게가 같지 않다. 질문은 독립 장면이 아니다. 질문→선택이 핵심 흐름이다. 결과는 안내가 아니라 획득이어야 한다. 기능은 의미를 만든다."

**변경**:
- 4컷(상황|질문|선택|결과) → **3컷(상황+질문 | 선택 | 결과+획득)**
- 질문을 상황에 통합: 만화에서 "고민하는 캐릭터"는 상황 컷 안에 있다
- 결과 패널에 **관점 카드 수집** 추가: 각 선택에서 한 줄짜리 관점을 카드로 획득
- 최종 리포트에 수집한 관점 카드 목록 표시

**디자인 변경**:
- 박스 라운딩 제거 → 직각
- 이미지 자리 → 정사각형 (ChatGPT 생성 이미지 삽입용)

**관점 카드 설계 원칙**:
- "기능은 의미를 만든다. 쓸일이 있으면 의미있다" — 피터공
- 관점 카드 = 이번 선택에서 배운 한 줄. 정보 확장, 어휘 전달도 가능
- 최종 리포트에서 "네가 모은 관점들"로 보여줌

## 왜 단일 HTML + 바닐라 JS인가

- 빌드 도구 없이 브라우저에서 바로 실행 가능
- 교육 현장에서 파일 하나만 배포하면 됨
- 원본 명세서 섹션 8-9에 React/TS/Vite 권장이 있었으나, 섹션 28의 "단일 HTML, 빌드 과정 없이 브라우저에서 열 수 있어야" 조건과 모순. 클라이언트 전용 원칙에 따라 단일 HTML로 통일

## 왜 서버 없이 시작하는가

- 1차 프로토타입은 핵심 루프 검증이 목적
- 서버 의존성은 배포/테스트 장벽
- 추후 로그/결과만 서버로 전송하면 됨
- trackEvent()를 미리 분리하여 나중에 fetch만 추가

## 왜 실제 AI API를 호출하지 않는가

- 프로토타입은 "AI 사용 시뮬레이션"이 아니라 "위임 판단 경험"
- API 비용/지연/인증이 교육 현장에서 장벽

## 왜 선택지에 정답을 만들지 않는가

- "정답이 있는 선택은 선택이 아니다" — 피터공
- 각 선택은 상황에 따라 합리적. 트레이드오프가 핵심
- "AI를 쓰지 마라"도 "AI를 써라"도 아닌, 판단의 근육을 기르는 경험

## 왜 trackEvent()를 미리 분리하는가

- 현재는 console + localStorage만
- 서버가 생기면 함수 내부에 fetch만 추가
- 분석 데이터 구조를 초기부터 확정

## 플레이테스트 확인 필요 사항

### ISSUE-1: 숫자 밸런스에서 정답이 보이는 문제

**현상**: "맡길 일" 선택은 항상 시간/에너지 소모 낮음 + 경험치 낮음. "내 생각/다시 보기"는 항상 소모 높음 + 경험치 높음. 두 시나리오만 지나면 "직접 하는 게 점수 많이 준다"를 학생이 눈치챈다.

**위험**: 트레이드오프가 아니라 최적화 문제가 됨. 기획문서의 "각 선택지는 나름대로 합리적이어야 한다" 원칙이 수치 레벨에서 깨짐.

**제안된 해법**:
- 자원 고갈 시 제약 도입 (시간 부족 → 빠른 선택만 가능)
- "맡길 일" 선택이 자원 보존 → 후반에 유리한 구조
- 시간/에너지 고갈 시 강제로 AI에게 맡길 수밖에 없는 상황

**플테 확인 질문**: "3번째 시나리오쯤 학생이 '항상 직접 하는 게 낫다'고 느끼는가?"

**결정 시점**: 플레이테스트 후

### ISSUE-2: 6개 시나리오로 성장감 발생 어려움

**현상**: 레벨업 임계값 Lv.2=30, Lv.3=70. 세 스킬을 골고루 선택하면 6시나리오 끝나도 어떤 스킬도 Lv.2에 못 닿을 수 있음.

**수학적 분석**:
- S03의 잠금 선택지는 delegateTask Lv.2 필요 → S01+S02에서 delegateTask만 골라도 10+8=18 < 30. **수학적으로 해금 불가**.
- S05/S06의 잠금 선택지는 reviewResult Lv.2 필요 → 2~3번 연속 reviewResult를 골라야 도달. 가능하지만 빡빡함.
- Lv.3(70)은 6시나리오 내에서 거의 불가능.

**위험**: 성장감 부재 + 잠금 선택지가 영원히 잠김.

**제안된 해법**: 임계값 하향(Lv.2=20), 복수 스킬 XP 증가, 잠금 선택지 Act3 배치

**플테 확인 질문**: "레벨업을 한 번도 경험하지 못하는 플레이가 발생하는가?"

**결정 시점**: 플레이테스트 후. DebugPanel에서 스킬 레벨 조작하여 잠금 해금 테스트 가능.

### ISSUE-3: 자원 고갈의 무의미

**현상**: 시간/에너지가 0이 되어도 아무 일도 안 일어남 (0에서 clamp만).

**위험**: 자원이 점수판일 뿐, 관리의 긴장감 없음.

**제안된 해법**: 시간 10이하 → 빠른 선택만 가능 등의 제약

**플테 확인 질문**: "시간/에너지가 줄어도 긴장감이 없는가?"

**결정 시점**: 프로토타입에서는 clamp만. 플레이테스트 후 제약 추가 여부 결정.

### ISSUE-4: 잠금 선택지 표시 방식

**현상**: "맡길 일 Lv.2 필요"라고 보여주면 "이전에 틀린 선택 했구나" 느낌을 줄 수 있음.

**대안**: 해금 시에만 "새 방법이 생겼다"로 표시.

**프로토타입 구현**: 현재는 잠금 상태 + 이유 표시로 구현 (명세서 기준). 플테 후 대안 검토.

## v1.1 성장 리포트 PDF 출력 방식 결정 (2026-05-13, 세션342)

> 진행 노트: [[요청.26.0513.0945-v11리포트독립화]]
> 관련 명세: v11/SPEC.md §13

### 채택안: window.print() 단일 옵션

학기 종합 리포트 하단 [리포트 저장 (PDF)] 버튼 → `printReport()` 함수 → `window.print()` 호출.

사용자 흐름: 클릭 → 브라우저 print 다이얼로그 → 학생이 "대상: PDF로 저장" 수동 선택 → 저장.

장점: 게임 본 폰트(Paperlogy) 그대로, 한글 정상, 출력 품질 깔끔, OS별 차이 없음, 의존성 추가 없음.
단점: 학생이 다이얼로그에서 한 단계 더 거쳐야 함 (원클릭 자동 다운로드 아님).

print 출력 전용 CSS: `src/styles/11-print.css` (`.no-print` 클래스로 화면 UI 숨김, 카툰 페이지 분할 제어, 배경/그림자 인쇄 활성화).

### 폐기된 시도 — 클라이언트 자동 PDF 다운로드 4종

원클릭 자동 다운로드를 위해 4가지 라이브러리/조합 시도, 모두 폐기.

| # | 시도 | 폐기 사유 |
|---|------|----------|
| 1 | html2pdf.js (html2canvas + jsPDF 번들, 캔버스 캡처) | 출력 품질 미달. 이미지·레이아웃 무너짐 |
| 2 | jsPDF + Paperlogy ttf (679KB, 게임 본 폰트) | 한글 글리프 빈 출력. 글자 자리만 차지 |
| 3 | jsPDF + NanumBarunGothic ttf (4MB, 검증된 한글) | 본 게임 통합 시점에도 한글 출력 실패 |
| 4 | jsPDF / pdfmake + 폰트 임베딩 없음 (시스템 기본) | 한글 깨짐. OS별(Windows/Android/iOS) 시스템 폰트 차이로 클라이언트 통일 불가 |

**근본 결론**: 클라이언트 PDF 라이브러리는 자기완결적 출력을 위해 폰트를 PDF에 임베딩해야 함. 한글 폰트 임베딩이 라이브러리별로 처리 방식이 다르고 글리프 매핑 실패 사례 다수. OS별 시스템 한글 폰트 차이로 클라이언트 단에서 통일된 한글 출력 어려움. 원클릭 자동 다운로드 + 영구 URL 요구가 발생하면 서버사이드 렌더링(Puppeteer 등)이 깔끔.

### 보류

- **Phase 2 서버사이드 PDF 저장**: 놀공 서버에서 리포트 데이터 POST → 서버가 HTML/PDF 생성·저장 → 영구 URL 반환. 현재 본 빌드 범위 외. 필요 발생 시 별도 결정.
