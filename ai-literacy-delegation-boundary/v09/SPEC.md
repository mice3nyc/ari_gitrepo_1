##### 0.3 v0.9 핵심 변경 상세

**자원 체계 전환**: §19에서 상세. 시간은 학기 총 예산(120, 회복 없음), 에너지는 집중력(70, 시나리오 후 고정 +18). AI를 쓰면 시간을 아끼고, 직접 하면 시간은 들지만 성장이 크다는 감각.

**비용-점수 분리**: 비용 숫자만 보고 정답을 추측할 수 없어야 한다. 점수는 선택의 의미로 판정(CSV에서 path별 직접 부여). AI 저비용이라도 자기 자료 + 검토하면 A~S 가능.

**피드백 2레이어**: §21에서 상세. `finals.shortFeedback`(결과 화면, 한 문장) + `finals.reportFeedback`(성장 리포트, 2~3문장). 기존 `finals.awareness` → `reportFeedback`으로 마이그레이션.

**용어 교체**: §22에서 상세. 위임판단력 → 판단하는 힘, 도메인지식 → 아는것의 힘. 교사 2인 공통 지적(초등 어려움) 반영. 5/8 피터공 확정.

##### 1.4 axisDelta — 카드 매칭 룰 데이터 (v0.7 재정의)

v0.6까지 axisDelta는 "시나리오별 카드 슬롯"(boostCard·bonusPoint)이었다. v0.7에서 이 자리를 폐기하고 **카드 매칭 룰 데이터**로 재정의한다. 어떤 19종 역량 카드를 보유한 상태에서 어떤 leaf에 진입할 때 +N 보너스가 붙는지 lookup 자리로 사용한다. §7.8 카드 매칭 활성화 메카닉의 데이터 백본.

```yaml
axisDelta:                          # v0.7 재정의 — 카드 매칭 룰
  - leafId: A1R3                    # 진입 leaf
    requireCard: 자기지식           # 보유해야 활성화되는 카드 라벨 (직접 매칭)
    bonusPoint: 5                   # 매칭 시 점수 보너스 (bonus 항으로 흡수)
  - leafId: B2R2
    requireCard: 비판적 사고
    bonusPoint: 5
  # 연관 매칭(역량 그룹) 룰은 v0.7.x 자리 — 그룹 yaml 정의 후 도입
```

**룰 두 결**:
- **직접 매칭** (v0.7 1차): `requireCard`와 학생 보유 카드 라벨이 정확히 일치
- **연관 매칭** (v0.7.x): 역량 그룹 yaml 정의 후. "자기지식" 보유 → "자기검증" 받는 leaf 활성화 같은 느슨한 매칭

폐기된 자리: `axisDelta.boostCard`, `axisDelta.bonusPoint`(시나리오 카드 슬롯), `triggerLeafIds`(역방향 lookup). 현재 leaf 기준 forward lookup으로 단순화.

##### 2.1 합산 식 (v0.7, v0.8에서 검수용 보존)

v0.6의 lookup 모델(`finals[leaf+review].score`로 27 자리 사후 lookup)을 폐기하고, 단계별 점수가 산식 그대로 쌓이는 합산 모델로 전환. 학생 화면에 산식이 그대로 도착하고 csv 검수 인터페이스가 직관 매치된다.

```
score = tier1.points + tier2.points + review.points + bonus
      = (베이스 20 + 변수 10) + (베이스 20 + 변수 10) + (R1=10/R2=20/R3=30) + (카드매칭 + 자원잔여)
      = 30 + 30 + 30 + 10  =  최대 100점
```

각 항의 자리 의미:
- **tier1.points = basePoint + varPoint** — 큰 방향(A·B·C)의 베이스 점수 + 옵션마다 박힌 변수 (시나리오당 3 자리 = 5 시나리오 15 자리)
- **tier2.points = basePoint + varPoint** — 행동 옵션의 베이스 + 변수 (시나리오당 9 자리 = 45 자리)
- **review.points** — 검토 깊이 단순 분배 (R1=10, R2=20, R3=30, 시나리오당 3 자리)
- **bonus** — 카드 매칭 보너스 + 자원 잔여 보너스 합쳐서 최대 10. 카드 매칭은 §7.8 메카닉, 자원 잔여는 v0.7.x 자리

폐기된 자리: `results[t2].basePoint`, `finals[leaf+review].score`(검수 참고용은 일정 기간 보존하되 코드 미사용), `useReviewLevelBoost`, `detected`, `levelStep`, `levelExtraBonus`, `getCardBonus`, `cardSlots`, `cardsHeld`. `getCardBonus`는 `getCardMatchBonus`로 재정의되어 bonus 항 안의 한 자리.

##### 2.2 yaml 스키마 — 단계별 점수


##### 3.3 시나리오 단위 누적 + 비대칭 효과 (5/3 세션278 root cause 정정)

**5/3 폭주 진단**: DELTA_POS={'++':4,'+':2} + competencyDiscountMult=2 → 실효 ×4. **그러나 root cause는 mult 값이 아니라 회로 — 한 시나리오 안에서 점수가 즉시 누적되어 다음 선택의 비용 효과가 즉시 변화**. 발란스 mult 조정으로는 폭주 강도만 줄어들 뿐 회로는 유지.

**root cause 정정 — 누적 단위를 시나리오로 끌어올림**:

- 한 시나리오 안에서는 비용 효과가 **고정** (학기 시작 ~ 이전 시나리오 끝까지 누적된 점수 기준)
- 학생의 선택은 **pending 점수**로 시각적으로만 표시 (§12 원 마커)
- 시나리오 끝에서 pending → 누적 게이지로 흡수
- 다음 시나리오부터 새 누적값으로 비용 효과 변화

**효과 식**:

```
displayed_cost = raw_cost - effect
effect = 누적_점수 × mult (시나리오 안 변화 X, pending 배제)
mult: 양수 ×2, 음수 ×2 (대칭 — 시나리오 단위라 폭주 위험 없음)
```

| 단계 | 정정 전 (5/3 박힘) | 시나리오 단위 + ±1 단순화 (5/3 최종) |
|---|---|---|
| 점수 매핑 | `{'++':4,'+':2}`/`{'-':-1,'--':-1}` | **선택당 ±1만** (`++/+ → 1`, `--/- → -1`) — 학생이 원 1·2·3개로 직관 |
| 효과 mult (양수) | 2 | 2 |
| 효과 mult (음수) | 2 | **1** (비대칭 — 잘못해도 한 배만 페널티, 세션289) |
| 누적 변화 시점 | 매 선택 | **시나리오 끝** |
| 한 시나리오 안 비용 효과 | 변화 (폭주 회로) | **고정** |
| 한 회기 max pending (양수) | — | **+3** (모두 +) |
| 한 회기 max pending (음수) | — | **-3** (모두 -) |
| DELTA_POS / DELTA_NEG 변수 | 사용 | **폐기** (`getAxisDelta(sign)` 함수로 단순화) |

CONFIG: `competencyDiscountMultPos: 2`, `competencyDiscountMultNeg: 1` (비대칭 — 양수 ×2 보상, 음수 ×1 페널티. 세션289 설정, v19 비용 하향 후 Pos ×2 복원 b6ed580).

코드:
- `_applyDiscount`는 `gameState.competencies.*.value` (이전 시나리오까지 누적) 기준 — pending 배제
- pending은 `gameState.pending.{delegation, knowledge}`에 박힘
- 시나리오 끝에 `absorbPending()` 호출 → value에 더하고 pending 0으로 reset

##### 3.4 공식형 박스 — 선택 화면 비용 표시 (v0.7 신설)

5/3 동현공 플테에서 도착한 결정적 발견을 화면 자리에 정정. ④ 할인 효과 메카닉이 학생 화면에 산식 그대로 도착하도록 두 줄 라벨 분리 박스로 표시. Cut2/Cut3 선택 화면에서 각 선택지 박스 아래에 다음 형식이 들어간다.

```
시간 비용 [10] − 위임 할인 [2] = 최종 시간 [8]
에너지 비용 [8] − 지식 할인 [3] = 최종 에너지 [5]
```

**박스 룰**:
- 두 줄 라벨 분리 — 시간 줄에는 위임판단력이, 에너지 줄에는 도메인지식이 할인 자리로 들어간다. 두 역량의 독립 학습 메시지가 살아남
- **최종 시간 / 최종 에너지 볼드** — 학생이 가장 먼저 읽어야 할 자리
- 학생 톤 — "시간 비용", "위임 할인", "최종 시간"처럼 한 번 읽고 이해 가능한 어휘
- 선택 화면 박스 정보 룰 (§7.8 박스 룰): 직관·구체만, 예상 점수 X
- **카드 매칭은 박스 자리 X** — 카드 매칭은 비용이 아니라 점수에 작용하므로 결과 패널(Cut6) 단계별 점수 자리에서 한 줄로 표시 ("자기지식 카드 매칭 보너스 +5")

**학생 화면 도착 검증**: 화면이 보내는 메시지 = 메카닉이 보내려는 메시지. "보유 위임판단력이 이 선택의 시간 비용을 -2 할인해줬다"가 박스 안 산식 그대로 학생 화면에 도착한다. v0.6에서 학생이 "선택지가 위·도를 깎는다"로 오해했던 자리 정정.

코드 자리: §6.6 참조 (`buildEffectCell`·`buildCostHTML` 공식형 박스 재작성).

**시각 위계 + 카드 레이아웃 재설계 (세션310)**: 선택지 카드 구조를 세로 바(A) | 본문 flex-row에서 검정 인라인 박스 + 선택 텍스트 / 비용 한 줄 flex-column으로 변경. 선택 텍스트 15px/700으로 격상. 비용 공식은 박스 제거 — 숫자만 색 글씨로 표시. 시간·에너지 두 그룹을 한 줄 좌우 배치, 사이에 #ccc 세로 구분선. 카드 풀 너비 활용으로 비용 공식이 잘리지 않음. 학생이 "무엇을 선택하는가"를 먼저 읽고, "얼마나 드는가"는 그 다음에 확인.

**v0.9 플테 피드백 반영 (세션319)**:
- **Cut2(1차 선택) 비용 제거** — 1차 선택은 비용 없이 순수 선택. 비용은 Cut3(2차 선택)와 Review부터 표시. 1차 선택지 전체 비용이 동일해 표시가 오히려 혼란.

##### 3.5 raw 비용 분배 룰 (v0.7 신설)

5/4 피터공 통찰: v0.6 raw 비용이 학생 직관과 어긋나는 자리가 있다. 모둠 발표에서 "AI 위임"(B)이 "다른 자료 기대기"(C)보다 시간 비용이 높게 박혀 있어, AI 사용의 일반 혜택(시간/에너지 절약)이 화면에 도착하지 않는다. 또 시간 비용이 에너지 비용보다 전반적으로 너무 높아 두 자원의 균형이 안 잡힌다.

**큰 룰 — 선택지 간 비용 위계**

| tier1 | 시간 비용 | 에너지 비용 | 근거 |
|---|---|---|---|
| **A 직접** | 가장 큼 | 가장 큼 | 자료 찾기 + 통합 + 작성 모두 본인 |
| **B AI 위임** | 가장 작음 | 가장 작음 (B1) ~ 큼 (B3) | AI 일반 혜택 = 절약. 검토·편집 강도에 따라 에너지 가변 |
| **C 기존 자료 기대기** | 중간 (A보다 작음) | 중간 | 찾고 통합하는 시간/에너지 듦. AI보다는 큼 |

**위계 규칙**: 같은 review 단계(R1)에서 `B시간 < C시간 < A시간`. AI(B)가 항상 가장 절약. 기존 자료(C)는 시간 절약은 되지만 에너지 절약은 약함.

**leaf 단위 시간 vs 에너지 분배 패턴**

leaf마다 디자인 의도에 따라 시간/에너지 비율이 다르다.

| 패턴 | 예시 leaf | 시간 | 에너지 | 근거 |
|---|---|---|---|---|
| 직접 + 깊은 작업 | A1, A2 | 매우 큼 | 큼 | 시간이 에너지보다 약간 큼 |
| 직접 + 가벼움 | A3 | 큼 | 중간 | 시간이 에너지보다 큼 |
| AI 풀위임 | B1 | 매우 작음 | 매우 작음 | 둘 다 절약 |
| AI + 일부 검토 | B2 | 작음 | 중간 | 시간 절약, 검토 에너지 듦 |
| AI + 본인 편집/통합 | B3 | 작음 | 큼 | "AI 쓰지만 글 대부분 편집" — 시간 ↓ 에너지 ↑↑ |
| 기존 자료 + 깊은 통합 | C1 | 중간 | 큼 | 통합 에너지 큼 |
| 기존 자료 + 보통 | C2 | 중간 | 중간 | 균형 |
| 기존 자료 풀의존 | C3 | 작음 | 작음 | 거의 그대로 사용 |

**review 가산**: R1=베이스, R2=+5 시간/+4 에너지, R3=+15 시간/+11 에너지 (현재 패턴 보존, 검토 강도가 비용 추가).

**stage floor (5/4 세션289 박음)**: t2와 review 단계는 시간 2 / 에너지 1 minimum. yaml resourceCosts는 leaf 총비용이고, formula가 t1=min, t2=min(t2family)-t1, review=leaf-t1-t2로 분해하는 구조라 가장 싼 t2 (A3·B1·C3)와 R1 그대로 제출이 0으로 도착했다. floor 박힌 자리: `_rawTier2Cost`·`_rawReviewCost` (코드) + `raw_t2_cost`·`raw_review_cost` (build_csv.py). yaml 자체는 변경 X. 가장 싼 leaf 한정 stage 합이 leaf 총비용보다 최대 4t/2e 큼 (overage 의도).

**시간/에너지 균형 룰**: 시간만 일률적으로 더 크지 않게. leaf 패턴에 따라 시간 우위(직접·AI 풀위임)와 에너지 우위(AI+편집·기존 자료 통합)가 갈린다.

**적용 범위**: 5 시나리오 × 27 자리(leaf+review) × 시간/에너지 = 270 자리 일괄 sweep. 시나리오 narrative와 leaf 디자인 의도 정합 검증.

**검수 자리**: 비용 sweep 완료 후 csv 리젠 → 피터공이 leaf 단위 검수.

##### 3.6 자원 비용 바닥값 (v14-slim 세션308)

후반 시나리오에서 역량이 높아지더라도 할인 후 비용이 0이 되면 선택의 무게가 사라진다.

```
displayed_cost = max(floor, raw_cost - effect)
floor = (raw_cost == 0) ? 0 : 1
```

- 원래 비용이 0인 선택지: 0 유지
- 원래 비용이 1 이상인 선택지: 할인 후 최소 1

코드: `_applyDiscount` 함수에서 `Math.max(1, discounted)` 적용. raw_cost 0 체크 선행.

```js
function getCardMatchBonus(leaf, ownedCards) {

  if (!rule) return 0;

}
```

**135 leaf 실제 분포** (`v07_시나리오_펼침_135행.csv` sweep, 자원 보너스 0 가정):

| 등급 | 카드 미보유 | 카드 매칭 시 |
|------|-----------|-------------|
| S | 0 (0.0%) | 5 (3.7%) |
| A | 5 (3.7%) | 0 (0.0%) ← 정점·정점·R3 자리에서 카드 매칭 시 95점 → S 점프 |
| B | 5 (3.7%) | 8 (5.9%) |
| C | 25 (18.5%) | 27 (20.0%) |
| D | 100 (74.1%) | 95 (70.4%) |

MIN 20점 / MAX 95점. **카드 매칭이 정점·정점·R3 자리(시나리오당 1자리 = 5)에서만 박혀 있어 카드 보유 시 A → S 점프**. 같은 카드를 두 번째 시나리오에서 매칭하면 S 도달 — 학습 의도(누적 학습 보상) 정합. 단, A가 0%인 분포는 학생 입장에서 A 등급 자체가 안 보일 수 있음 (csv 콘텐츠 리뷰 자리).

D 안 점수 폭(20~69, 50점 폭)이 넓어 회피·부정 자리들이 변별 가능. 회피/부정 + R1 = 20~30, 회피/부정 + R3 = 40~55, 적절·중립 + R2 = 50~65.

**학기 종합 리포트**(`showFinalReport`, line 2988)는 등급 매핑 안 사용. 4유형(주도형/분별형/의지형/출발형/탐색형) + 총점 + 카드 누적만. 등급은 시나리오 단위(Cut6 결과 패널)만.

#### 7. 역량 카드 시스템 (5/3 신설, boostCard 라벨 22종 폐기)

##### 7.1 컨셉

기존 review 결과 `boostCard`(시나리오 특화 메시지 22종 — "처음의 나 카드", "타인의 말 경고 카드")는 **폐기.** 같은 자리(시나리오 끝 도메인 지식 양수)에 **역량 카드**를 획득하는 시스템으로 대체. 역량 카드는 학생이 이해 가능한 일반 역량 라벨(문해력·협업능력·발표력 등)을 사용하고, 같은 라벨이 여러 시나리오에서 반복 획득 가능 → 학습 메시지 강조 효과.

> "도메인 지식"은 학생에게 어려운 용어. 역량 카드는 시나리오 안에서 구체 역량 라벨로 표현된다. "도메인 지식(문해력)" 식의 표기는 게이지 라벨에서만 사용, 카드는 후자만.

##### 7.2 도메인 풀 — 공통 + 시나리오 특화 (5/3 세션282 확정)

##### 7.4 획득 조건 + 카드 데이터 구조

- **조건**: 시나리오 끝 시점 도 축 점수가 양수(>0)이면 그 leaf의 `competencyCards` 배열 카드 모두 획득

##### 7.5 인벤토리 UI (슬라이드 패널)

- **탭 버튼**: 화면 우측 끝 고정 (세로 라벨 "📇 카드" 또는 "역량 카드"). 클릭 시 패널 슬라이드 인/아웃
- **패널 폭**: 320px 가량, 우측에서 슬라이드. 닫기 버튼 + 배경 dim
- **카드 표시 (5/3 갱신)**: 카운트 + 라벨 가로 배치
  - 카운트: 큰 볼드 + 배경색 강조 + 좌측 배치 (font-size 23px·font-weight 800·color #111·배경 #f0f0f0·padding 2px 10px·min-width 36px·radius 4px)
  - 라벨: 카운트 우측 (15px·700·카드 색상 — `_cardColor(label)` 활용)
  - 정렬: 카운트 내림차순 → 라벨 가나다순 (`.sort((a,b) => groups[b].length - groups[a].length || a.localeCompare(b, 'ko'))`)
  - "×N" 표기 폐기 — 카운트는 단독 숫자로
  - 클릭 시 시나리오·leaf 출처 expand
- **빈 상태**: "아직 획득한 카드가 없어요. 시나리오 끝에 카드를 받을 수 있습니다."

##### 7.6 reward 팝업 — 단건 컨펌 (5/3 세션278 재작성, cascade·skip 폐기)

- **트리거**: 시나리오 끝 chain의 [1] 단계. 도 축 양수 + `competencyCards` 배열 비지 않음
- **연출 (단건)**:
  1. 카드 1장 화면 중앙 fadein (0.5s) — 라벨 + note + **"획득" 버튼**
  2. 학생이 "획득" 클릭 → 우측 탭 버튼 위치로 travel (0.6s) + scale 축소 → 사라짐
  3. 다음 카드 자동 fadein (간격 0.2s buffer)
  4. 마지막 카드 travel 끝나면 chain 다음 단계로 자동 진행
- **인벤토리 badge**: travel 끝나는 시점부터 탭 버튼 unread badge 노출
- **`note` 텍스트**: 카드 본문에 표시 ("자기 경험을 직접 고르고 끝까지 확인해서 이후 자기표현 카드와 잘 맞음")
- **폐기**: cascade 자동 진입(2s 간격), 스킵 버튼, ESC 단축키, 배경 클릭 닫기 — 모두 제거
- **이유**: 학생이 카드가 무엇인지 + 획득했다는 사실을 강조. 모방·회피로 흐를 수 있는 자동 연출 폐기. 컨펌 = "이 카드를 내 것으로 받았다"는 능동적 자리

**카드 못 받은 회기**: cut6 결과 패널에 음수 안내 박스(`"이번엔 역량 카드를 받지 못했어요"`) — 도 축 음수 || competencyCards 0장일 때 출력. 카드 reward 단계는 건너뜀.

##### 7.7 위 축(위임 판단력) 표현 — 별도 결정 큐 (5/3 세션282 갱신)

위 축은 카드로 만들지 않음. 점수 + 게이지로만 표시 (현재 구조 유지). 측정 가능·객관 기준으로 학생에게 전달하는 방법은 후속 결정 자리:
- 옵션 A: 위 축도 카드로 (e.g. "위임 결정 ★", "AI 사용 검토 ★") — **미해결**
- 옵션 B: 시나리오 끝 위 축 결과 메시지 + 누적 그래프 (학기 끝 라벨에서 단계적 노출) — **미해결**
- 옵션 C: 학기 끝 4유형 라벨에만 (지금 그대로) — **5/3 종합 리포트에 카드 누적 박혀서 옵션 C 일부 선택된 상태** (학기 끝 4유형 + 위/도 카드 누적 그리드)

5/3 결정 시점은 1차 빌드 + 플테 후 → 별도 사이클. 옵션 A·B는 여전히 미해결.

###### 7.8.2 카드 매칭 활성화 룰

학생이 학기 동안 누적한 카드(`gameState.inventory.competencyCards`)와 진입할 leaf의 매칭 룰(`axisDelta` 재정의 데이터, §1.4)을 lookup. 매칭 시 두 자리에 효과가 들어간다.

2. **결과 패널 한 줄** — Cut6 결과 패널 단계별 점수 자리에 "자기지식 카드 매칭 보너스 +5" 한 줄로 표시 (§6.6.4)

###### 7.8.3 박스 정보 룰 — 선택 화면 vs 결과 패널

학생 화면의 두 자리는 다른 룰을 따른다.

| 자리 | 룰 | 표시 가능 |
|---|---|---|
| **선택 화면 (Cut2/Cut3)** | 직관·구체만, 예상 점수 X | 공식형 박스 비용·할인·최종 (§3.4) |
| **결과 패널 (Cut6)** | 사후 노출, 점수·매칭 보너스 직접 표시 | 단계별 점수 + narrative + 카드 매칭 한 줄 |

카드 매칭 활성화 신호는 결과 패널 자리에 들어간다. 선택 화면 박스에는 카드 매칭 X (학생이 선택 시 매칭 정보를 보면 정답 풀이가 되어 학습 메시지 흐려짐). 사후에 "내가 키운 역량이 다음 선택을 가능하게 했다"는 누적 메시지로 도착시킨다.

#### 11. 타이틀+튜토리얼 화면 (5/3 세션278 신설)

##### 11.1 컨셉

게임 진입 시 곧바로 시나리오 선택 화면(`showStartScreen`)으로 가는 흐름을 폐기. 그 위에 **타이틀+튜토리얼 한 화면**(`showTitleScreen`)을 신설한다. 학생이 메카닉 무게(두 축·자원·카드·레벨업·4유형)에 들어가기 전에 한 번 펼쳐 보고 진입하는 자리.

> 한 화면 안에 타이틀 + 한 줄 명제 + 핵심 메카닉 2~3불릿 + "시작" 버튼 — 스크롤 없이.

##### 11.2 진입 흐름

```
페이지 로드
  ↓
gameState.tutorialSeen ?
  ├─ true  → showStartScreen()         (시나리오 선택)
  └─ false → showTitleScreen()         (타이틀+튜토리얼)
                ↓ "시작" 버튼 클릭
                gameState.tutorialSeen = true; saveGame()
                ↓
                showStartScreen()
```

- **첫 진입에만**: tutorialSeen 플래그가 false일 때만 자동 노출
- **다시 보기**: showStartScreen 화면에 "튜토리얼 다시 보기" 링크 → 클릭 시 showTitleScreen 호출 (플래그 변경 X)

##### 11.3 gameState 확장

```js
// createInitialState() 안
tutorialSeen:false   // 첫 시작 버튼 클릭 시 true
```

#### 12. pending 시스템 + 원 마커 (5/3 세션278 신설, §3.3 짝)

##### 12.1 컨셉

한 시나리오 안의 점수 변화를 "확정 누적"이 아닌 "임시 pending"으로 표시. 학생은 매 선택마다 원 마커로 즉각 피드백 받지만, 비용 효과 계산은 이전 시나리오 끝의 누적값으로 고정. 시나리오 끝에서 pending → 누적 흡수.

**해결하는 문제**:
- 한 시나리오 안 비용 효과가 매 선택마다 변하는 회로(×4 폭주의 root cause) 폐기
- 그러면서도 학생의 선택 피드백은 원 마커로 살림 — 두 시간 축 분리

##### 12.2 gameState 확장

```js
// createInitialState() 안
pending:{delegation:0, knowledge:0}   // 시나리오 시작 0, 끝에서 value에 흡수
```

continueGame guard: `if(!gameState.pending)gameState.pending={delegation:0,knowledge:0};`

| 함수 | 정정 전 | 정정 후 |
|---|---|---|
| `applyTier1` | `competencies.delegationChoice.value+=delta` | `pending.delegation+=delta` |
| `applyTier2` | 동일 | `pending.delegation+=delta` |

##### 12.4 원 마커 UI (5/3 후속 — 좌우 분할, 5/9 세션320 인라인 이동)

- **위치**: stat-header 행 안에 인라인 배치 — `[라벨] [pending-dots] [숫자]` 한 줄. pending-dots 컨테이너를 좌우 50/50 분할 (`.pending-dots-neg` + `.pending-dots-pos`). 라벨과 같은 줄에 놓아 패널 높이 절약
- **양수 pending**: 중앙에서 **우측**으로 초록 원 N개 (좌측에서 우측으로 뻗음)
- **음수 pending**: 중앙에서 **좌측**으로 빨강 원 N개 (우측에서 좌측으로 뻗음, `flex-direction:row-reverse`)
- **0**: 양쪽 모두 빈 상태
- **양수 → 음수 전환**: 초록이 깨지는 연출 (0.3s) + 빨강 등장
- **음수 → 양수 전환**: 빨강이 사라지고 초록 등장
- **CSS**: `.pending-dots` 컨테이너 + 좌우 분할 자식(`.pending-dots-neg`, `.pending-dots-pos`), `.pending-dot.positive`/`.negative`, `.pending-dot.breaking` keyframe
- **이유**: 게이지 자체가 좌(음)/우(양) 방향성. 원 마커도 같은 방향성 채택 → 색 + 방향 두 채널로 직관 강화. 한 시나리오 안 pending max ±3이라 좌/우 50% 공간 충분
- **유의**: SPEC 라벨 -4/+8은 점수 max 값(단위당 폭 비대칭)이지 시각 자리 아님. 게이지 zero선 = 시각 50%

##### 12.5 시나리오 끝 흡수

cut6 chain의 첫 단계로 박음 — 카드 reward 직전:

```
[0] 결과 패널 (cut6) — 1.4초
        ↓
[1] pending 흡수 — pending dot이 게이지로 흘러들어가는 애니메이션 (0.8s)
        - value+=pending → 게이지 갱신
        - pending=0 → dot 클리어
        ↓
[2] 카드 reward (도 양수+카드 있을 때)
[3] 레벨업
[4] 자원 충전
```

`absorbPending()` 함수: `competencies.*.value += pending.*` → `pending.* = 0` → setBipolar 갱신 + 마커 클리어 + 흡수 애니메이션 재생. Promise 반환.

##### 13.2 인간중심 역량 카드 — 3축 + 12태그

**축 1 — 중심잡기**: AI 시대에 자기 기준과 목표를 붙잡는 힘
| 태그 | 의미 |
|---|---|
| 주체성 | 명확한 목표와 기준을 가지고 AI 활용을 주도하며 결과에 책임지는 힘 |
| 적응성 | 새로운 AI 기술과 환경 변화를 유연하게 받아들이고 활용하는 힘 |
| 호기심 | AI 답에 멈추지 않고 더 묻고 탐색하는 태도 |

**축 2 — 융합하기**: AI와 내 생각을 엮어 문제를 풀거나 새 결과를 만드는 힘
| 태그 | 의미 |
|---|---|
| 창의적 사고 | 내 생각과 AI 도움을 엮어 새로운 결과를 만드는 힘 |
| 문제해결적 사고 | 문제의 핵심을 파악하고 AI를 어디에 쓸지 정하는 힘 |

| 통합적 사고 | 여러 정보와 관점을 연결해 균형 있게 정리하는 힘 |
| 맥락적 사고 | AI 결과를 상황·목적·사람에 맞게 해석하는 힘 |

**축 3 — 성찰하기**: AI 결과와 나의 선택을 다시 확인하고 책임 있게 고치는 힘
| 태그 | 의미 |
|---|---|
| 비판적 사고 | AI 결과의 오류·빈틈·편향을 의심하고 따져보는 힘 |
| 윤리적 사고 | 내가 하지 않은 일을 한 것처럼 쓰지 않고 책임 있게 AI를 쓰는 힘 |
| 성찰적 사고 | AI를 쓴 과정과 내 선택을 되돌아보고 더 낫게 바꾸는 힘 |
| 사회·관계적 사고 | AI가 낀 상황에서도 타인과 협력하고 존중하며 소통하는 힘 |

인벤토리 누적은 3종(중심잡기 N장 / 융합하기 N장 / 성찰하기 N장). 펼치면 세부 태그별 카운트.

#### §14. 리플레이 시스템 (v0.8 신설)

##### 14.1 컨셉 (세션310 재설계)

B 이하 결과를 받은 학생이 회복력 카드를 통해 같은 시나리오를 다시 해볼 수 있다. 카드 자체가 리플레이 진입점.

- **회복력**: B 이하 결과 시 자동 지급. 화면 중앙에 특별 카드로 등장. "회복역량 사용해 시간 되돌리기. 시나리오 다시 도전" 클릭 → 리플레이 진입. 카드를 사용하지 않고 다음으로 넘어갈 수도 있음.
- **도전력**: 리플레이 완료 + 이전 대비 개선 시 지급. 리플레이 끝 chain에서 다른 역량카드처럼 팝업으로 획득 확인.

##### 14.2 결과 화면 흐름

**B 이하 결과 시**:
1. 결과 제목 + 최종점수 + 등급 표시
2. CUT6 보정 피드백 표시
3. 획득 카드 (인간중심/도메인 — B, C만)
4. **회복력 카드 — 화면 중앙 특별 등장** (점선 테두리 + ↺ 심볼 + 어두운 톤)
   - "회복역량 사용해 시간 되돌리기. 시나리오 다시 도전" 버튼
   - "다음 시나리오로" 버튼 (사용하지 않는 선택지)
5. 회복력 인벤토리 반영

**A 이상 결과 시**:
1. 결과 제목 + 최종점수 + 등급
2. CUT6 보정 피드백 + 획득 카드
3. `리플레이 제안` 문구 표시
4. "다음 시나리오로" 버튼

**리플레이 완료 시**:
1. 결과 화면 (위와 동일)
2. 점수 향상 시 → **도전력 카드 팝업** (점선 테두리 + ↑ 심볼) 다른 카드와 함께 표시
3. 도전력 인벤토리 반영

##### 14.3 리플레이 로직

```
리플레이 시작 시:
  1. 현재 자원 상태를 스냅샷 (해당 시나리오 시작 전 상태로 복원)
  2. 해당 시나리오의 gameState 초기화 (선택·점수·카드)
  3. 시나리오 첫 컷으로 이동

리플레이 완료 시:
  1. 새 최종점수/최종등급 확인
  2. 이전 최종등급 또는 최종점수보다 개선되었으면 → 도전력 지급
  3. best score 유지 (이전 vs 현재 중 높은 점수)
  4. 도전력은 해당 시나리오에서 1회만 (중복 지급 없음)
```

##### 14.4 gameState 확장

```js
gameState.replay = {
  // 시나리오별 리플레이 상태
  [scenarioId]: {
    played: false,          // 리플레이 해봤는가
    improved: false,        // 개선했는가 (도전력 지급 조건)
    bestScore: 0,           // 최고 점수
    bestGrade: '',          // 최고 등급
    resourceSnapshot: null  // 시나리오 시작 전 자원 스냅샷
  }
}
```

##### 14.5 학기 종합 리포트에서

- 리플레이한 시나리오는 best score 기준으로 표시
- 성장 카드 섹션: 회복력 N장 + 도전력 N장
- "다시 도전해서 더 나은 선택을 했어요" 같은 성장 narrative

##### 15.3 이중 클릭 방어

모든 진행 버튼(다음, 선택, 다시 해보기 등)에 debounce 적용. 클릭 후 300ms 내 재클릭 무시. v0.7 P1-3 흡수.

```js
function safeClick(btn, handler) {
  let lastClick = 0;
  btn.addEventListener('click', function(e) {
    var now = Date.now();
    if (now - lastClick < 300) return;
    lastClick = now;
    handler(e);
  });
}
```

##### 15.4 결과 카드와 리포트 일관성

결과 화면에서 보여준 카드와 최종 리포트의 카드 인벤토리는 같은 데이터 소스(gameState)를 참조. 결과 화면에서 카드를 gameState에 추가 → 리포트에서 gameState를 읽어 표시. 불일치 발생 시 gameState가 진실.

##### 17.3 패턴별 성장 요약 문장

texts.yaml `growthReport.patterns`에 5패턴 문장 저장:

##### 17.4 강점: 카드명 기반 직접 매핑

reportStrengthTags 대신 **획득한 카드의 texts.yaml short 설명을 직접 사용**. 빈도 매칭 없이, 학생이 실제로 얻은 카드 하나하나의 의미를 보여준다.

강점 데이터 흐름:
```

  → TEXTS.humanCentricCards[axis].tags[tag].short
  → "명확한 목표와 기준을 가지고 AI 활용을 주도하며 결과에 책임지는 힘"

finals[leaf].domainCards[i]
  → TEXTS.domainCards[name].short
  → "결과를 다시 확인하고 고치는 힘"
```

보완은 기존 reportGrowthTags → improvements 풀 매칭 유지 + C/D 시나리오의 reportReflection을 보충 표시.

#### §18. 학기 종합 리포트 통합 재설계 (세션308)

§6.4의 기존 6단계 구조를 폐기하고, 학생이 "나는 어떤 학습자인가?"를 한 군데서 읽도록 통합한다.

##### 18.1 폐기하는 것

- **상단 4박스 중 위/도 2칸**: "위임 판단력 +10", "도메인 지식 +15" — 내부 계산값, 학생에게 의미 없음
- **4유형 의미 박스**: "위 +10 / 도 +15 AI를 동료로..." — 내부 수치 노출
- **_reportNarrative 함수**: 학습자 패턴, 카드 누적 인사이트, 선택 패턴 — 성장 리포트에 통합

##### 18.2 새 구조 (showFinalReport)

```
1. 타이틀 + 부제
2. 2박스 통계 (총점 / 최종 레벨)  ← 위/도 제거
3. 성장 리포트 (통합 — _renderGrowthReport)
   a. 학습자 유형 (Aware Builder 등 + 유형 설명)
   b. 패턴별 성장 요약
   c. 내가 키운 역량 (카드명 + short 설명)
   d. 카드 분포 (인간중심 N장 / 도메인 N장 / 성장 N장)
   e. 학생 선택 패턴 (A/B/C 분포 + 해석)
   f. 더 연습할 과정
   g. 다음에 이렇게 해볼 수 있다
   h. 교사용 관찰 포인트 (접기)
4. 카드 그리드 (역량 카드 목록)
5. 카툰 돌아보기
6. "학기 처음부터" 버튼
```

##### 18.3 _reportNarrative → _renderGrowthReport 통합

기존 _reportNarrative의 세 블록을 _renderGrowthReport 안으로 이동:

| 기존 블록 | 통합 위치 | 변경 |
|-----------|----------|------|
| 학습자 패턴 (4유형 + 유형명 + 설명) | 성장 리포트 3a | 유형명/설명 유지, "위/도" 수치 레이블 제거 |
| 카드 누적 인사이트 (top 카드 + 분포 + 0장 회기) | 성장 리포트 3c~3d | "내가 키운 역량" 뒤에 분포 한 줄 추가 |
| 학생 선택 패턴 (tier1 A/B/C 분포 + 해석) | 성장 리포트 3e | 그대로 이동 |

##### 18.4 상단 통계 박스

```
[총점 420]  [Lv.5]
```

2칸만. 위/도는 디버그 패널에서만 확인 가능.

#### §19. 자원 체계 v20 전환 (v0.9 신설)

v0.8까지의 자원은 시간·에너지 100/100 시작, 등급별 차등 회복이었다. v20에서는 두 자원의 성격을 명확히 분리한다.

##### 19.1 시간 — 학기 총 예산

```js
time.current = 120
time.max = 120
```

시간은 시나리오 종료 후 회복하지 않는다. AI를 쓰면 시간을 아낀다는 감각이 살아야 한다. 5개 시나리오를 모두 직접 수행으로만 통과하기는 어렵지만, AI 활용과 직접 수행을 섞으면 충분히 완주 가능해야 한다.

##### 19.2 에너지 — 집중력

```js
energy.current = 70
energy.max = 70
```

시나리오 종료 후 고정량 회복:

```js
energy.current = Math.min(energy.max, energy.current + 18)
```

회복량은 등급과 무관하게 고정. 이유: 잘했을 때만 많이 회복되면 부익부 구조가 강해진다.

**v0.8 회복 체계 폐기 대상**:
- `recoverBase: 0.45` → 제거
- `gradeBonus: { S: 0.35, A: 0.25, B: 0.15, C: 0.10, D: 0 }` → 제거
- 등급별 총 회복률(S 80%~D 45%) → 고정 +18로 교체

##### 19.3 1차 선택 비용 균일화

모든 1차 선택 비용:

```js
tier1Cost = { time: 1, energy: 1 }
```

1차 선택은 방향 선택이지 실제 수행이 아니다. 비용 차이가 크면 학생이 숫자로 정답을 추측한다.

##### 19.4 단계별 비용 기준 (v20 데이터)

**더 자세히 선택 (tier2)**:

| 선택 유형 | 시간 | 에너지 |
|---|---:|---:|
| AI 대행 | 3~4 | 2~3 |
| AI 시작 도움 | 5~7 | 4~5 |
| AI 질문/구조화 도움 | 7~9 | 5~7 |
| 내 정보/자료 기반 AI 활용 | 9~12 | 7~10 |
| 직접 핵심 수행 | 14~18 | 12~16 |
| 비효율/회피성 수행 | 10~16 | 10~15 |

**검토 선택 (review)**:

| 검토 유형 | 시간 | 에너지 |
|---|---:|---:|
| 그대로 진행 (R1) | 1~2 | 1~2 |
| 간단 검토 (R2) | 4~6 | 3~5 |
| 깊은 검토 (R3) | 8~11 | 7~9 |

실제 값은 v20 costs CSV에서 leaf별로 직접 부여. 위 범위는 데이터 설계 가이드라인.

##### 19.5 v20 CSV 구조

**content CSV** (5 시나리오 × 27 leaf = 135행, 51컬럼):
코드에서 사용하는 값: `최종점수`, `최종등급`, `결과화면 짧은 피드백`, `성장리포트 상세 피드백`, `획득 역량카드`, `획득 인간중심역량`, `획득 도메인카드`, `획득 성장카드`

6. [3] 레벨업 모달 (에너지 자동 충전, RP 분배 없음)

##### 19.7 도전력 판정 수정 (세션316 버그 수정)

`startScenario`에서 `replay[scid]`를 미리 생성하므로, 결과 화면에서 "이미 존재 → 리플레이"로 오판정되는 버그 수정. 덱스 `12-ari-challenge-card-bug.md` 권장안 적용.

**판정 구조 (wasReplay + challengeAwardedNow):**

```js
var replayState = gameState.replay[scid];
var wasReplay = replayState && replayState.played === true;
var challengeAwardedNow = false;

if (!replayState) {
  // 방어 — startScenario에서 보통 이미 생성됨
  replayState = { played:true, improved:false, bestScore, bestGrade };
} else if (!wasReplay) {
  // 첫 플레이 완료 — played를 true로 전환, 도전력 없음
  replayState.played = true;
  replayState.bestScore = score;
  replayState.bestGrade = grade;
} else {
  // 진짜 리플레이 — score 비교 후 도전력 조건부 지급
  if (score > replayState.bestScore && !replayState.improved) {
    replayState.improved = true;
    challengeAwardedNow = true;
    inventory.growthCards.push({label:'도전력',...});
  }
}
```

카드 애니메이션에서 `challengeAwardedNow`로 도전력 표시 제어.

#### §20. 할인 이중 구조 (v0.9 신설)

v0.8의 할인은 축 점수(위임→시간, 도메인→에너지) 단일 체계였다. v0.9는 축 숙련도 할인 + 카드 태그 매칭 할인의 이중 구조로 전환한다.

##### 20.1 축 숙련도 할인 (기존 계승, 폭 축소)

위임/도메인 축 점수는 유지한다. 학생의 전반 성장 상태를 나타내고, 게임 안에서 레벨업/숙련도 체감을 만든다.

변경:
- 기존 할인 로직 유지하되 할인 폭을 v20 밸런스에 맞게 축소
- 시간 할인 상한: 원래 시간 비용의 **20%**
- `competencyDiscountMultPos` 값 재조정 필요 (플테 후 튜닝)

##### 20.2 카드 태그 매칭 할인 (v21 데이터)

역량카드가 선택지와 매칭될 때 **에너지만** 할인한다. 시간은 카드로 줄이지 않는다.

선택지 데이터 구조 (`discountTags` in scenarios.yaml):

```yaml
discountTags:
  humanCentric: "성찰하기"       # 인간중심 축태그 (3종: 중심잡기/성찰하기/융합하기)
  domain: ["검토력", "표현력"]    # 도메인 카드명
  strongDomain: ["검토력"]        # 강한 매칭 도메인 카드명 (에너지 -3)
```

매칭 규칙:
1. `humanCentric` 축태그와 보유 인간중심 카드의 axis 일치 → **-2**
2. `domain` 태그와 보유 도메인 카드 label 일치 → 카드당 **-2**
3. `strongDomain` 태그와 보유 도메인 카드 일치 → 카드당 **-3** (domain의 -2를 대체)
4. 한 선택에서 카드 할인 합계 최대 **-6**
5. 같은 카드 여러 장이어도 1회만 적용

데이터 소스: `v21-card-discount-tags.csv` (135 path × 3단계)

##### 20.3 합산 및 상한

```js
rawTimeCost = stageCost.time
rawEnergyCost = stageCost.energy

axisTimeDiscount = calculateAxisTimeDiscount(choice, competencies)
axisEnergyDiscount = calculateAxisEnergyDiscount(choice, competencies)
cardEnergyDiscount = calculateCardEnergyDiscount(choice, inventory)

totalTimeDiscount = axisTimeDiscount
totalEnergyDiscount = axisEnergyDiscount + cardEnergyDiscount

maxTimeDiscount = Math.floor(rawTimeCost * 0.2)
maxEnergyDiscount = Math.floor(rawEnergyCost * 0.4)

appliedTimeDiscount = Math.min(totalTimeDiscount, maxTimeDiscount)
appliedEnergyDiscount = Math.min(totalEnergyDiscount, maxEnergyDiscount, 6)

finalTimeCost = Math.max(1, rawTimeCost - appliedTimeDiscount)
finalEnergyCost = Math.max(1, rawEnergyCost - appliedEnergyDiscount)
```

##### 20.4 UI 표시

할인이 적용될 때 두 출처를 구분해서 보여준다:

```text
숙련도 적용: 에너지 -2
문해력 카드 적용: 에너지 -3
```

학생이 느껴야 할 것: "내가 전반적으로 성장해서 부담이 줄었다 + 특정 카드가 이번 선택에서 도움을 줬다."

v0.8의 공식형 박스(§3.4) 형식은 유지하되, 카드 할인 줄을 추가한다.

##### 20.5 v0.8 폐기 대상

- `competencyDiscountMultPos: 2` → 축소 (구체 값은 플테 후)
- `competencyDiscountMultNeg: 1` → 축소 또는 유지
- 시간 할인에 축 점수만 사용 (카드는 시간 미할인)

#### §21. 피드백 2레이어 (v0.9 신설)

v0.8의 `finals.awareness` (CUT6 보정 피드백) 단일 문장을 두 레이어로 분리한다.

##### 21.1 결과 화면용 — shortFeedback

시나리오 종료 직후 CUT6 결과 카드에 표시. 한 문장 중심.

```js
finals.shortFeedback
```

예: "내가 고른 단어에서 출발해 글 안에 내 기억이 남았어요."

##### 21.2 성장 리포트용 — reportFeedback

학기 종합 리포트와 상세 성장 리포트에 사용. 2~3문장 허용.

```js
finals.reportFeedback
```

예: "내가 고른 단어에서 출발해 글 안에 내 기억이 남았어요. 다만 바로 넘겨 어색한 연결은 확인하지 못했어요."

##### 21.3 마이그레이션

기존 `finals.awareness` → `finals.reportFeedback`으로 이동. `finals.shortFeedback`은 v20 CSV에서 새로 부여된 값 사용.

코드에서의 사용:
- CUT6 결과 화면: `finals.shortFeedback` 표시
- §17 성장 리포트 / §18 학기 종합 리포트: `finals.reportFeedback` 표시

#### §22. 용어 교체 (v0.9 신설)

교사 피드백(최서연+사성진 선생님): "위임", "도메인" 등의 단어가 초등 수준에서 어렵다. 5/8 피터공 확정.

##### 22.1 교체 목록

| 기존 용어 | 새 용어 | 적용 범위 |
|---|---|---|
| 위임 판단력 / 위임판단력 | **판단하는 힘** | 학생 화면, 튜토리얼, 카드, 리포트 |
| 도메인 지식 / 도메인지식 | **아는것의 힘** | 학생 화면, 튜토리얼, 카드, 리포트 |

##### 22.2 적용 범위

**교체 대상** (학생에게 보이는 모든 곳):
- 튜토리얼 4문장 (§11.5)
- 선택지 비용 공식형 박스 (§3.4 → §20.4)
- CUT6 결과 패널 카드 이름
- 성장 리포트 카드 목록 / 유형 설명
- 학기 종합 리포트 통계 라벨
- texts.yaml 내 카드 description / narrative / 메시지

**유지 대상** (내부 변수):
- 코드 변수명 (`competencyDiscount`, `axisDelta` 등) — 내부 변수는 기존 유지 가능
- yaml 키 (`delegation`, `domain`) — 데이터 호환성
- SPEC 내부 참조 — 명확성을 위해 "판단하는 힘(위임축)", "아는것의 힘(도메인축)"으로 병기

##### 22.3 튜토리얼 갱신

§11.5의 튜토리얼 4문장에서 "위임판단력", "도메인지식"을 새 용어로 교체. 추가 설명 한 줄:
- 판단하는 힘: "무엇을 맡기고 무엇을 직접 할지 따져보는 힘"
- 아는것의 힘: "내가 가진 정보, 자료, 경험이 결과를 만드는 힘"

#### §23. 선택지 인라인 전개 — 컷 흐름 재설계 (세션320)

##### 23.1 문제

기존: "어떻게 할까?" 클릭 → 이미지 없는 빈 Cut 2 패널에서 선택지가 뜸. 맥락(도입 텍스트)과 선택이 다른 패널에 분리되어 있어 학생이 맥락을 잃고 선택한다.

##### 23.2 새 흐름

선택지는 **현재 패널 본문 아래에 펼쳐지고**, 선택 후 **다음 패널이 이미지+결과와 함께 활성화**된다.

| 단계 | 패널 | 내용 | 버튼 |
|------|------|------|------|
| 도입 | Cut 1 | 상황 텍스트 | "어떻게 할까?" |
| 1차 선택 | Cut 1 (하단 펼침) | tier1 선택지 카드 | — |
| 1차 결과 | Cut 2 (활성화) | 이미지 + 선택 요약 | "더 자세히 — 어떻게 할까?" |
| 2차 선택 | Cut 2 (하단 펼침) | tier2 선택지 카드 (비용 포함) | — |
| 2차 결과 | Cut 3 (활성화) | 이미지 + 선택 요약 | "결과 확인하기" |
| 결과 | Cut 4 (활성화) | 결과 텍스트 + 점수 | "검토할까?" |
| 검토 선택 | Cut 4 (하단 펼침) | review 선택지 카드 (비용 포함) | — |
| 검토 결과 | Cut 5 (활성화) | 이미지 + 검토 요약 | — |
| 최종 | Cut 6 (활성화) | 등급 + 피드백 | 다음/리플레이 |

##### 23.3 함수 매핑

| 기존 | 변경 후 | 역할 |
|------|---------|------|
| `goCut2()` | `showTier1Choices()` | Cut 1 body에 선택지 append |
| `goCut3()` | `showTier2Choices()` | Cut 2 body에 선택지 append |
| `goCut5()` | `showReviewChoices()` | Cut 4 body에 선택지 append |
| `onTier1()` | (수정) | fadeOut → Cut 2 활성화 (요약+버튼) |
| `onTier2()` | (수정) | fadeOut → Cut 3 활성화 (요약+버튼) |
| `goCut4()` | (수정) | "결과 확인하기" 클릭 시. 버튼이 `showReviewChoices` 호출 |
| `onReview()` | (수정) | fadeOut → Cut 5 활성화 (요약) + goCut6 |
| `goCut6()` | (수정) | Cut 5 요약 설정 제거 (onReview에서 처리) |
| `fadeOutChoices()` | (수정) | fade 후 choices-area를 DOM에서 제거 — 패널 크기 복원 |
| `continueGame()` | (수정) | 복원 chain을 새 흐름에 맞게 |
| `dbgJumpCut()` | (수정) | 디버그 chain 갱신 |

##### 23.4 핵심 원칙

- 선택지가 있는 곳 = 맥락이 있는 곳. 빈 패널에서 선택하지 않는다
- 선택 후 다음 패널이 이미지와 함께 "결과"로 도착한다
- `fadeOutChoices` → 패널 전환의 자연스러운 전이 유지
- 6패널 그리드 구조(3+3)는 유지. 콘텐츠 배치만 변경
