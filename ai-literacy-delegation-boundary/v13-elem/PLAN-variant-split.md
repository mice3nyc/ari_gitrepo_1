## PLAN — 초등/중등 변종 분기 준비 (단일 코드 × 변종 콘텐츠)

> 분기 방식 확정(피터공 6/17): **단일 코드 × 변종 콘텐츠.** 코드(`src/`)는 한 벌 공유, 콘텐츠(`data/*.yaml` + 이미지)만 변종별로 가른다. 빌드는 `build.py`가 src + data를 합쳐 단일 index.html 산출.
>
> 지금 단계: **분기 시작 아님.** 분기를 위한 준비를 전부 현재 중등 빌드(v13-mid)에서 끝낸다. 이 노트는 그 준비의 근거(코드 감사)와 작업 목록.
>
> 선례: `_dev/DMZ_v5/` — `shared/index_base.html` 하나에서 build.sh가 플래그로 mobile/offline/sequential 3변종 산출. AI리터러시도 같은 골격으로 이식.

### 근본 진단 — 이중 진실원(dual source of truth)

콘텐츠 데이터(시나리오 키·카드 카테고리·축·학습자유형)가 **yaml에도 있고 코드에도 박혀 있다.** 콘텐츠는 yaml로 갈릴 수 있게 돼 있지만, 코드에 박힌 사본이 중등을 가정하므로, 변종이 갈리면 코드 사본이 깨진다(대부분 조용한 실패 — 빌드는 통과하고 런타임에 오작동).

**분기 준비의 실체 = 코드에 박힌 콘텐츠 가정을 전부 yaml/CONFIG로 내려보내기(외부화).** 이걸 중등에서 다 해두면, 초등은 yaml만 갈아끼우면 됨.

증거: v12-elem(5/17 통째 복사)은 한 달 만에 src/js 17개 중 9개가 v13-mid와 갈라지고 신규 기능 2개 누락 → 죽은 빌드. 통째 복사 방식의 실패 표본.

### ★ 피터공 5축 너머 발견 — 콘텐츠 차이가 메카닉까지 번지는 지점

피터공이 든 변종 축: 시나리오·이미지·카드명칭·카드종류·카드개수. 감사 결과, **콘텐츠 차이가 콘텐츠 yaml에 머물지 않고 게임 메카닉/판정 코드까지 흔드는** 지점 셋이 드러남:

1. **카드 개수 → 할인·레벨 밸런스 (H7).** 할인액과 HUD 레벨이 "보유 카드 장수"로 계산됨. 초등이 카드를 적게/다르게 주면 **밸런스가 통째로 달라짐.** 이건 콘텐츠가 아니라 밸런스 설계 변수.
2. **학습자 유형 6종 → 판정 코드 (H8).** 유형 개수·판정 기준이 코드에 박힘. 초등이 더 단순한 유형 체계를 쓰면 yaml 교체가 아니라 코드 변경.
3. **게임 트리 구조** — 6컷 / tier1 3분기(A/B/C) / 검토 R1·R2·R3 / 시나리오당 깊이가 엔진에 고정. 변종이 이 뼈대를 공유하면 무관, **안 하면 엔진 자체를 변종화해야 함(작업 규모 10배).**

→ **분수령 질문**: 초등이 게임의 뼈대(트리 구조 + 할인/레벨 메카닉 + 학습자유형 체계)를 중등과 공유하는가?
- 공유 → 분기 준비 = 콘텐츠 외부화(위 H/M 목록). 관리 가능.
- 비공유 → 엔진/메카닉도 변종화. 별도 설계 필요.

### 결정 반영 (피터공 6/17)

**뼈대 공유 확정** — 게임 트리 구조(6컷 / tier1 A·B·C / review R1·R2·R3 / 시나리오 깊이) 안 바꿈. **엔진 변종화 없음.** 분기 준비 = 콘텐츠 외부화로 한정.

**다이나믹 요소는 둘** (나머지는 전부 사전 고정 데이터):
- 할인 = 보유 카드 장수 (시간=위임칸 카드수 / 에너지=능력칸 카드수)
- **RP(자원 충전 보너스)** = 매 시나리오 컷6에 `awardRP` 적립. CONFIG 전부 데이터 주도(양호, 값만 변종화):
  - `rpRewardByGrade`: S30/A20/B15/C10/D5
  - `rpLevelUpBonusByLevel`: Lv2+10/3+15/4+20/5+25 (그 시나리오 레벨업 시만)
  - 적립 = 등급RP + 레벨업RP (두자리수). 사용 = `rpCost.time/energy=1`, 1RP=시간/에너지+1, 플레이어 분배
  - ⚠️ 후속 후보(분기 무관): `awardRP`가 `balance=total` **덮어쓰기** → 안 쓴 RP 다음 시나리오에 소멸. 리셋 의도면 정상/누적 의도면 버그. 피터공 판단 대기.

**학습자 유형 분기 전략 — "텍스트만 분기" 채택 + 전제 1개:**
- 유형을 **판정 규칙**(분포→유형키)과 **표현 텍스트**(이름·거울문장·설명) 두 층으로 분리
- 판정 규칙 = 공유 (런칭 시 초등/중등 동일 6종 참조)
- 표현 텍스트 = 변종별 분리
- 이유: 런칭(동일6종+텍스트만 다름)은 텍스트 분리로 충분. 나중에 유형 개수·판정이 갈리면 변종 폴더에 규칙 override 얹으면 됨(없으면 공유 fallback). 미래 비용 0.

**★ 변종 시스템 설계 원칙 — "공유 default + 변종 override":**
모든 변종 변수(학습자 유형 / RP 밸런스 / 할인 산식 / 카드 세트)에 일관 적용.
- 공유 default를 `content/_shared/`에, 변종 차이만 `content/{elem,mid}/`에
- 변종 파일에 키가 있으면 그게 이김, 없으면 공유 default
- "모든 조건을 미리 준비"하지 않고 "지금 다른 것만 분리 + 나중에 갈리면 그때 override 추가" → 피터공 "최소 상황 대응" 원칙 충족

### 로그 시스템 합류 (SPEC-play-log 흡수, 6/17)

[[SPEC-play-log]] 흡수 결과 — 충돌 없음, 변종 분기와 정합. 연동 지점:
- **`v`(=CONFIG.version) 필드가 변종 식별자.** 값에 이미 "mid" 포함(`v1.3-mid-r39`) → 초등은 `elem`. 통합 분석 시 변종 구분 자동, 별도 태그 불필요.
- 로그 철학 "코드만 저장 + 게임 데이터 조인" = 변종별 데이터 조인과 정합.
- **변종 변수 2개 추가 확정**: ① `outboxKey`(현 `...-v13-mid-outbox` 하드코딩 → 변종별 분리, storageKey 분리 원칙) ② `CONFIG.version`(H2 CONFIG 변종화와 동일 작업).
- **공유 1개**: 학습자 유형 판정 함수 = 리포트·로그·게임이 한 곳 공유(H8 외부화 자리). 판정 로직 분산 금지.
- additive 설계라 로그 구현과 외부화 리팩토링 상호 독립 진행 가능.
